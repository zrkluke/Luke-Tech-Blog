## 下集：打破全表掃描，GIN 索引的終極魔法

### 5. EXPLAIN 揭露的殘酷真相
首先我用 PostgreSQL 的 `EXPLAIN ANALYZE` 跑了一下執行計畫，希望從數據支持。
當使用多重 `LIKE` 時，執行計畫裡出現了可怕的巢狀迴圈（Nested Loop），CPU 確實為了比對不同的標籤，在同一列資料上來回折返跑了無數次。

相較之下，Regex 的確乾脆俐落多了，固定就是一次運算。但這就夠了嗎？

### 6. 暴風雨前的寧靜：五千萬筆資料的恐懼
雖然 Regex 減輕了 CPU 的負擔，但真正的危機才正要浮現。
這個專案的資料量**預估會落在 5,000 萬到 1 億筆之間。**

在這種天文數字的資料量下，光是把 1 億筆資料從硬碟裡撈出來（I/O 全表掃描），可能就要花上好幾分鐘。就算 CPU 算得再快，硬碟讀取速度的物理極限依然會把整支程式拖垮。**如果不能降低 I/O 負擔，我們注定會在這場大數據戰役中慘敗。**

### 7. 降維打擊：Array 轉型與 GIN 索引抽屜
既然不能一筆一筆找，那就得幫資料建「目錄」。而在 PostgreSQL 裡，處理多重標籤的終極神兵利器，就是 **GIN（通用倒排索引）**。

**複雜度上發生什麼事？** 

沒有可用索引時，不論是多重 `LIKE` 還是 Regex，要對「每一列是否符合條件」下結論，往往仍被 **總筆數 $N$** 綁在 **$O(N)$** 等級的掃描與字串工作上（上集的 **$O(N \times M)$** 則再多乘一層條件數 $M$）。改成在 **Array 欄位上建 GIN** 之後，對「陣列是否包含某標籤」這類條件，優化器通常改走 **GIN 條目反查**：先在索引樹裡找到該標籤（步數會隨索引變大而變長，但大約是「對數級」那種慢慢變長，不像掃表那樣一筆加一筆），再取出對應的列指標（TID）集合。**真正要從表裡讀回來的**，多半是「條件命中的那些列」，而不是整張表；因此實際 I/O 與後續運算，會從「接近整表 $N$」轉成「主要由**索引查找 + 命中筆數**決定」。多條件時常見 **BitmapAnd / BitmapOr**，把比對從「逐列字串」換成「位元圖集合運算」，進一步避開被 $N$（或 $N \times M$）主導的形態。

**和上集對照：** 

把多重 `LIKE` 收成 **Regex**，多半仍走在同一條 **$O(N)$ 掃描** 上，只是把「每一列上的 CPU」從約 **$O(M)$** 次字串比對，壓成約 **單次** 狀態機掃描——屬於**少掉一個 $M$ 乘數**、或說**常數級變好**的層次。**改用索引**則是換了一條演算法路徑：**不再用總筆數 $N$ 當主導項**，搜尋空間從「整表」縮成「索引鍵 + 命中集合」。因此在複雜度意義上，**索引帶來的鬆綁通常遠大於「只把 LIKE 改成 Regex」那一級**；Regex 救 CPU，索引才是在大表上救 I/O 與整體量級的關鍵。

**實務上怎麼做?**

我決定對資料結構動刀，進行兩步大改造：

**第一步：將字串打散成陣列 (Array)**

利用 `string_to_array()` 函數，把原本黏在一起的字串 `'tag1;tag2'`，轉型成 PostgreSQL 原生支援的陣列格式 `{"tag1", "tag2"}`。範例（假設舊欄位叫 `tags`，分隔符為 `;`）：

```sql
-- 新增陣列欄位
ALTER TABLE customers ADD COLUMN tags_array text[];

-- 將既有字串灌進陣列（空字串用 nullif 避免變成含空元素的陣列）
UPDATE customers
SET tags_array = string_to_array(nullif(trim(tags), ''), ';');
```

**第二步：建立 GIN 倒排索引**

為這個全新的 Array 欄位加上 GIN 索引。

```sql
CREATE INDEX CONCURRENTLY idx_customers_tags_gin
  ON customers USING gin (tags_array);
```

查詢時可改用陣列運算子（例如 `tags_array && ARRAY['tag_VIP']::text[]` 表示有交集、`@>` 表示包含），讓優化器走 GIN，而不再依賴全文掃描的 `LIKE`／Regex。

**這個動作徹底改變了資料庫找資料的邏輯！**
它不再把一筆一筆的客戶資料拿出來看，而是反過來，把所有的標籤變成一個個的**「抽屜」**（就像這篇文章的封面圖那樣）。

從結構上看，**GIN 把資料依標籤分組、聚合的效果，和「倒排索引」是一樣的**：索引裡的鍵是 **tag（詞項）**，值是 **出現該 tag 的列指標（TID）**——就像搜尋引擎用關鍵字反查文件 ID，而不是逐篇全文掃描。換句話說，查詢條件裡「要找哪個標籤」這種 **查詢意圖**，被直接對應到一小撮候選列；不必再把意圖拆成對每一列的字串比對迴圈。

*   `tag1` 抽屜打開 ➔ 裡面放著包含 1 號、5 號、99 號客戶的 ID 名單。
*   `tag_VIP` 抽屜打開 ➔ 裡面放著 2 號、8 號客戶的 ID 名單。

當下次前端要尋找 `tag_VIP` 時，資料庫不必再為了這個條件去掃完整張 1 億列的表；它直接走到 `tag_VIP` 的抽屜前，抽出裡面的名單，再依 TID 精準回表取資料。

對應到前面的複雜度直覺：**主導成本從「被整表 $N$ 綁死的掃描」**，變成「先在索引裡對準標籤（查找步數約隨索引規模對數成長），再只對**真正命中的那些列**做回表／合併」——當命中筆數遠小於總筆數時，體感才會像「從 $O(N)$ 被降維」。

### 結語
從一開始簡單無腦的 `LIKE` 比對，到思考 CPU 負擔的 `Regex` 比對，再到最終解決 I/O 物理極限的 `Array + GIN Index`。這次的優化經驗讓我深刻體會到：

> **效能優化不是只看語法層面的對錯，而是要深入理解底層資源（CPU vs I/O）到底被消耗在哪裡。**

唯有用對資料結構，才能讓資料庫發揮它真正的威力！
