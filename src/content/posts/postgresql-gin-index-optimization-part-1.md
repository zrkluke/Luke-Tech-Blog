## 上集：當 LIKE 遇上 Regex 的觀念碰撞

### 1. 前情提要
故事的開端，源自於工作上承接的一批 ETL（資料萃取、轉換、載入）大數據資料。
在前端或資料處理的階段，為了方便儲存，客戶的多重屬性標籤常常被塞在同一個文字欄位裡，並用分號 `;` 隔開。

資料的長相大概像這樣：
`tag1;tag2;tag3;tag_VIP;tag_inactive`

因為資料來源已經是 ETL 處理過的「完成品」，我也沒想太多，理所當然地就順著這個結構繼續往下寫 SQL 查詢了。

### 2. 同事的靈魂拷問：用 Regex 比較快？
當我在寫標籤過濾條件，準備用一堆 `LIKE '%tag1%' OR LIKE '%tag2%'` 來找出特定客群時，旁邊的資深同事看了一眼，給了一個建議：
> 「嘿，你要不要把這些條件合併，改用 Regex（正則表達式）來寫？這樣效能會比 LIKE 好喔！」

### 3. 我的第一反應：這不都是全表掃描嗎？
聽到這個建議，我當下其實是在心裡打了一個大大的問號。
我的直覺邏輯是：在沒有建立任何索引的情況下，不論是 `LIKE` 還是 Regex，資料庫不都得乖乖地從第一筆資料掃描到最後一筆嗎？

如果時間複雜度同樣都是 $O(N)$（也就是必須遍歷 $N$ 筆資料），那改用 Regex 真的會比較快嗎？

### 4. 破除迷思：我們都對，只是看到了不同的瓶頸
經過一番討論與推敲，我發現**我們兩個的想法都是對的，只是我們關注的「效能瓶頸」完全不同。**

*   **我的想法（I/O 負擔）：** 我滿腦子想的都是「硬碟讀取」。既然沒有索引，資料庫就必須把海量資料全部從硬碟搬進記憶體，這個巨大的 **I/O 成本**，確實不管是 LIKE 還是 Regex 都無法解決。
*   **同事的想法（CPU 負擔）：** 他看到的是「運算成本」。若用多個 `LIKE` 加上 `OR` / `AND` 串起 **$M$** 個標籤條件，對 **$N$** 筆資料而言，最直覺的實作往往是「每一列、每一個條件都要比一次」，字串比對次數可粗估為 **$O(N \times M)$**。若改寫成**單一一條** Regex，正則引擎會編譯成狀態機，對每列字串通常只需**單次通關掃描**就能判斷真假，規模約 **$O(N)$**（仍受樣式長度與回溯影響，但至少不再多乘一層條件數 $M$），因而大幅降低了 **CPU 負荷**。

原來，在硬碟 I/O 注定悲劇的前提下，同事的 Regex 確實能有效拯救快要燒起來的 CPU！
